# 배경

2026년 4월 1일, Claude Code [이하 `클로드 코드`] 의 소스 코드가 유출되었다. map 파일은 배포용으로 난독화된 자바스크립트 코드를 복원하는 용도인데, 이 map 파일이 npm 레지스트리를 통해 누구나 접근할 수 있게 되었다. (클로드 코드의 Boris Cherny 는 개발자 실수로 인한 사고라고 트위터에서 밝혔다.)

아직 출시되지 않은 미공개 기능이나 내부 보안 시스템까지 드러나게 된 셈인데. 너무나 안타까운 소식이지만 ... 군소 개발자로서 LLM 하네스 엔지니어링 노하우를 얻어갈 수 있는 기회라고 생각하고 스터디를 시작했다. "Opus, Sonnet 등 모델 자체의 우수성도 있겠지만, 다른 곳보다도 클로드가 훨씬 *알잘딱깔센*을 잘하는 이유가 무엇일까?" 라는 궁금증도 어느 정도 해결할 수 있으리라 기대해본다.

# 목표

- Harness Engineering [이하 `하네스 엔지니어링`] 개념을 이해한다
- 클로드 코드의 설계 철학을 이해하고, 나의 문제 상황에 적용할 수 있을지 고민한다

# 하네스 엔지니어링  

- 원래 내가 이해하고 있던 방식 : AI 인스턴스 단위에 역할을 정의하고, 맥락 정보를 잘 주고 받을 수 있도록 디자인하는 시스템. 
- OpenAI 블로그 글로 정의한 방식 : 사람이 직접 코드를 작성하는 대신, **에이전트가 안정적으로 작업을 수행하도록 환경·구조·피드백 루프를 설계하는 엔지니어링 방식**

원래 작년부터 '하네스'라는 키워드가 알음알음 쓰였던 것 같은데, OpenAI 에서 [Harness engineering: leveraging Codex in an agent-first world](https://openai.com/ko-KR/index/harness-engineering/?ref=blog.risemoment.ai)라는 글을 공개하며 '하네스 엔지니어링'이 현 시점의 메타가 되었다.

글에서는 엔지니어의 역할이 개발자에서 시스템/환경 설계자로 역할이 전환되었다고 짚고 있다. 사람은 이제 코드를 작성하지 않고,
- 작업 의도 정의
- 에이전트가 갖동하는 환경, 도구, 구조 설계
- 피드백 루프 구축  
... 을 담당하게 되었다는 것이다.

그 이유는 "모델 자체의 성능"보다 "모델이 참고할 수 있는 문서나 규칙"이 부족하다는 문제 상황 때문이다. 이제는 AI 자체가 아니라 AI 에이전트가 보고 이해할 수 있는 지식 시스템 (knowledge system)이 관건이 되었다. 에이전트가 할 일이 작업 단위로 잘 분해되어 있고, 이들이 참고할 수 있는 실행 가능한 지식이 갖추어져 있고, 검증 시스템이 자동화되어 있다면, 사람은 방향을 제시하며 최종적인 판단만 담당한 채 개발 프로세스를 에이전트에게 위임할 수 있다는 개념이다.


## 원래 내가 AI 를 사용하던 방식

사실, 이렇게 개념만 보았을 때는 잘 와닿지 않는다. 내가 AI를 활용하던 방식과 비교했을 때라야 감이 올 것 같다.

나에게 익숙한 화면은 두 가지다. chatGPT 와 같은 일대일 상호작용 UI. 그리고 한편에 코드를 띄워 놓고 다른 한편의 채팅UI 를 통해 읽고 쓰기 작업을 진행하는 Cursor.

추가로 최근에 '멀티 에이전트 시스템'의 개발을 담당하게 되어, ADK/LangGraph 기반의 파이프라인을 구축하기도 했다. ADK Visual Designer 나 LangSmith Studio 등으로 접근할 수도 있지만, 기본적으로는 코드 베이스로 실행하는 형태다.
```
python main.py
```

세 방식은 모두 활용 목적이 상이하다. 하지만 모두 "AI 의 출력 결과물만을 소비하는 데 그친다" 라는 점을 공유한다.
- chatGPT : 일상, 업무 환경에서의 질의응답
- Cursor : 코드 작성 (파일 실행 권한은 주지 않았다)
- LangGraph : 업무 니즈에 딱 부합한 입출력 데이터를 자동 생산

하네스 엔지니어링에서는 AI 가 행동 주체로서 역할하기 때문에 내가 그것을 컨트롤한다는 점이 포인트다.
